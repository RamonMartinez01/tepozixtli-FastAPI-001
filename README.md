# Hñäki-app (Core Engine: Tepozixtli)
### Asynchronous satellite telemetry motor for food security and agricultural decision making.

*Project born at the HackathonLAC Copernicus 2026.*

### The Problem: The Data Gap
In the field of agriculture and food security, information is the primary line of defense. Terabytes of high-precision satellite data generated by the Copernicus program (Sentinel-1 and Sentinel-2) exist, but these resources are inaccessible to those who need them most: farmers. The barrier is not a lack of data, but rather the computational capacity to process it and the need to translate it from raw variables into actionable agronomic language at the right time.

### The Solution: Hñäki-app
Hñäki-app is a platform designed to democratize access to geospatial intelligence. Powered by its internal Tepozixtli engine, the application establishes an asynchronous pipeline capable of ingesting satellite telemetry on demand. It transforms complex variables—such as Land Surface Temperature (LST), Soil Moisture, and Vegetation Indices (NDVI)—into efficient, ultra-fast loading visual representations (Cloud Optimized GeoTIFFs).El objetivo principal es entregar a los productores del campo una herramienta que procese matemáticas complejas en la nube, entregando respuestas puntuales y actualizadas directamente en sus navegadores, empoderando así la toma de decisiones en el territorio.

## Backend Architecture
At the core of this repository is a microservices-oriented architecture designed with a strict separation of concerns: a synchronous flow for public consumption, and a protected asynchronous flow for heavy telemetry ingestion.


(![Tepozixtli Architecture Diagram]())

### Flow 1: Public Consumption (Synchronous & Lightweight)
End-user interactions via the client application (Hñäki-app) are completely isolated from the computational cost of the Copernicus API.

1. **Query:** The user filters the map by polygon, date, and indicator (e.g., NDVI).
2. **Rapid Response:** FastAPI queries the PostgreSQL/PostGIS database to check for the existence of this data intersection.
3. **Delivery:** If the information was previously processed, the backend responds in milliseconds with the public URL of the Cloud Optimized GeoTIFF (COG) hosted in the cloud, allowing the client to render the map layer immediately.

### Flow 2: Ingestion Engine (Asynchronous Admin Layer)
Heavy geospatial processing is strictly reserved for an internal administration panel (CMS). This flow allows managers to audit information gaps and trigger satellite data collection missions at the project's discretion, preventing resource abuse and mitigating automated traffic (bots).

1. **Authorization (FastAPI):** An RBAC-protected endpoint receives the administrator's command to process a new polygon and date. FastAPI extracts the geometry from PostGIS, delegates the task, and returns an HTTP 202 Accepted.
2. **The Buffer (Redis):** Acts as the message broker, queuing the payload to ensure operational stability.
3. **The Ingestor (Go Worker):** An independent, high-concurrency process that intercepts the task:
   * Authenticates and requests raw telemetry from the **Copernicus Programme API**.
   * Executes the mathematical and geospatial transformation of the raw file (TIFF) into a high-performance format **(Cloud Optimized GeoTIFF - COG)**.
4. **Deployment (DigitalOcean Spaces):** The worker transfers the processed map to an S3-compatible Object Storage and retrieves the public CDN URL.
5. **Final Handshake (Webhook):** The worker notifies FastAPI of the successful mission. FastAPI then updates the PostGIS catalog, making the new map immediately available for Flow 1 (Public Consumption).

## Engineering Decisions & Trade-offs
Designing a system capable of handling heavy geospatial data on constrained resources (1 vCPU, 2GB RAM) required strict architectural boundaries. Here is the rationale behind our core technical decisions:

### Polyglot Architecture: Python (FastAPI) + Go
  * **The Challenge:** Processing raw satellite imagery is highly CPU and memory-intensive. Handling this within Python's Global Interpreter Lock (GIL) would block the event loop, causing API timeouts and potential Out-Of-Memory (OOM) crashes on our constrained server.
  * **The Solution:** We adopted a polyglot approach. Python (FastAPI) is strictly used for web orchestration, ORM integration (PostGIS), and rapid request handling. The heavy lifting is delegated to a standalone worker written in Go. Go’s compiled efficiency, low memory footprint, and robust concurrency model keep the background processing stable.
### Format Strategy: Cloud Optimized GeoTIFFs (COG)
  * **The Challenge:** Traditional raster files (Raw TIFFs) require the client browser to download the entire dataset (often 10MB+ per layer) before rendering, leading to massive bandwidth costs and browser crashes on lower-end devices.
  * **The Solution:** We explicitly pay an upfront computational cost in the backend. The Go worker translates the raw Copernicus data into Cloud Optimized GeoTIFFs (COG). This format natively supports HTTP Range Requests.
  * **The Trade-off:** Slower backend ingestion times, but it allows the client application (React) to request only the specific pixels needed for the user's current zoom level and viewport. The result is zero client-side memory overload and lightning-fast map rendering.
### Infrastructure Vision: Digital Sovereignty & Vendor Neutrality
  * **The Challenge:** Relying on proprietary serverless ecosystems (e.g., AWS Lambda, Step Functions) can lead to severe vendor lock-in and unpredictable billing spikes.
  * **The Solution:** The entire stack is containerized using Docker and orchestrated via docker-compose. We deploy on independent virtual machines (DigitalOcean Droplets) paired with an agnostic S3-compatible Object Storage.
  * **The Trade-off:** We assume the responsibility of manual infrastructure orchestration and OS-level maintenance, but we guarantee digital sovereignty, 100% predictable costs, and the flexibility to migrate the entire stack to any cloud provider—or even on-premise bare metal—in minutes.

## Technical Challenges Overcome
Deploying a multi-container asynchronous processing engine requires navigating complex networking and resource constraints. Key challenges successfully mitigated in production include:

### Resource Optimization on Constrained Infrastructure

* **The Context:** The entire production stack (PostgreSQL/PostGIS, Redis, FastAPI, and the Go Worker) runs on a single, entry-level Virtual Machine (1 vCPU, 2GB RAM).
* **The Challenge:** Geospatial raster processing involves generating large ephemeral files (Raw TIFFs and intermediate COGs). Without strict management, these temporary files would rapidly exhaust the container's disk space and memory, causing system failure.
* **The Resolution:** Implemented rigorous container hygiene. The Go worker acts as a precise garbage collector, ensuring that every raw TIFF and temporary COG is securely purged from local storage the millisecond the S3 upload succeeds. This guarantees a stable, near-zero footprint on the server's disk over time.

### Dynamic CORS & Reverse Proxying
* **The Context:** Serving the API to a secure Single Page Application (React) hosted on Netlify.
* **The Challenge:** Hardcoding absolute URLs for Cross-Origin Resource Sharing (CORS) creates rigid, insecure environments and triggers browser-level "Mixed Content" blocks when transitioning from HTTP (backend) to HTTPS (frontend).
* **The Resolution:** Refactored FastAPI’s CORS middleware to dynamically ingest allowed origins via environment variables, completely decoupling the codebase from the client layer. Paired this with a reverse proxy strategy at the edge (Netlify), masking the backend behind a secure route (/api/*) and entirely bypassing strict browser restrictions.
